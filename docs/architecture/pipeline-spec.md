# Pipeline Specification

> Extraído das regras do `runner.pipeline.md` do OpenSquad, estendido para o OrbitMind.

---

## Visão Geral

O pipeline é a sequência de etapas que um squad executa para completar uma tarefa. Cada etapa é associada a um agente e pode ser executada inline (no contexto do runner) ou como subagent (subprocesso independente).

---

## Pipeline YAML

```yaml
# pipeline/pipeline.yaml
name: "Pipeline de Marketing"
steps:
  - id: step-01
    name: "Pesquisa de Mercado"
    agent: researcher
    execution: subagent
    model_tier: fast
    file: steps/step-01-research.md
    outputFile: output/research.md

  - id: step-02
    name: "Estratégia de Conteúdo"
    agent: strategist
    execution: inline
    model_tier: powerful
    file: steps/step-02-strategy.md
    outputFile: output/strategy.md

  - id: step-03
    name: "Aprovação da Estratégia"
    type: checkpoint
    message: "Revise a estratégia e aprove para continuar."
    options:
      - "Aprovar e continuar"
      - "Solicitar revisão"
      - "Cancelar pipeline"

  - id: step-04
    name: "Criação de Copy"
    agent: copywriter
    execution: inline
    model_tier: powerful
    file: steps/step-04-copy.md
    outputFile: output/copy.md
    depends_on: [step-02]             # OrbitMind extension

  - id: step-05
    name: "Design Visual"
    agent: designer
    execution: inline
    model_tier: powerful
    file: steps/step-05-design.md
    outputFile: output/slides/
    parallel_with: [step-06]          # OrbitMind extension
```

## Fases de Execução

### Fase 0: Inicialização

1. Carregar `squad.yaml`, company.md, memories.md
2. Ler `pipeline/pipeline.yaml` para definição dos steps
3. **Resolver skills:** verificar instalação e configuração MCP
4. Gerar run ID: `YYYY-MM-DD-HHmmss` (com collision check)
5. Criar diretório de output: `output/{run_id}/`
6. Inicializar estado: todos os agentes `idle`

### Fase 1-N: Para Cada Step

**Início do step:**
1. Atualizar estado: agente → `working`
2. Ler arquivo do step: `steps/{step-id}.md`
3. Checar modo de execução

**Se `type: checkpoint`:**
- Apresentar mensagem ao usuário
- Se tem `options`, aguardar escolha (numbered list)
- Salvar resposta no `outputFile`
- Avançar ao próximo step

**Se `execution: subagent`:**
- Checar `model_tier` (default: powerful)
- Carregar definição do agente
- Aplicar Output Path Transformation
- Despachar subagente com contexto completo
- Aguardar conclusão e verificar output

**Se `execution: inline`:**
- Assumir persona do agente
- Anunciar: `{icon} {Agent Name} está trabalhando...`
- Seguir instruções do step
- Salvar output

### Handoff (Entre Steps)

```
1. Agente atual: status → "delivering", deliverTo → "{next_agent}"
2. Aguardar 3 segundos (animação UI)
3. Agente atual: status → "done", deliverTo → null
4. Próximo agente: status → "working"
```

### Conclusão

1. Salvar output final
2. Estado: `status: completed`, todos agentes `done`
3. Atualizar memória do squad (learnings, preferences)
4. Apresentar resumo de conclusão

---

## Output Path Transformation

```
Passo 1: Inserir run_id
  output/research.md → output/2026-03-21-143022/research.md

Passo 2: Inserir versão (apenas se Step 1)
  output/2026-03-21-143022/research.md → output/2026-03-21-143022/v1/research.md
```

---

## Veto Conditions

Steps podem definir condições que invalidam o output de um agente:

```markdown
## Veto Conditions
- Output tem menos de 500 palavras
- Não cita pelo menos 3 fontes
- Contém informação factual não verificada
```

**Processo:**
1. Avaliar cada condição contra o output
2. Se alguma acionada: informar e pedir correção (max 2 tentativas)
3. Após 2 falhas: apresentar ao usuário para decisão

## Extensões OrbitMind (vs. OpenSquad)

### Execução Paralela
```yaml
- id: step-04
  parallel_with: [step-05, step-06]  # Rodam ao mesmo tempo
```

Steps listados em `parallel_with` executam concorrentemente. O pipeline só avança quando todos completam.

### Steps Condicionais
```yaml
- id: step-07
  condition: "step-03.output == 'Aprovar e continuar'"
  skip_message: "Estratégia rejeitada, pulando criação"
```

### Dependências Explícitas
```yaml
- id: step-05
  depends_on: [step-02, step-03]     # Aguarda ambos completarem
```

### CI/CD Integration
```yaml
- id: step-10
  type: ci
  action: github.create_pr
  config:
    repo: "org/repo"
    branch: "feature/{run_id}"
    title: "Auto-generated by {squad_name}"
```

### Retry Policy
```yaml
- id: step-04
  retry:
    max_attempts: 3
    backoff: exponential
    delay_ms: 1000
```

---

## Tratamento de Erros

| Cenário | Ação |
|---------|------|
| Subagente falha | Retry 1x, depois oferece skip/abort |
| Step file ausente | Sugerir edição do squad |
| Company.md vazio | Redirecionar ao onboarding |
| Budget excedido | Pausar execução, notificar |
| Timeout | Cancel após 10 minutos, notificar |
